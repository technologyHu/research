一句话读懂

Meta-Harness 把 Agent 外围的提示、检索、记忆、工具调用、完成检查等运行时逻辑当成可搜索的代码，让编码智能体读取所有历史候选的源码、分数和执行轨迹，再提出下一版 harness。

做 Agent 的人越来越容易遇到一个现象：同一个基础模型，放进不同的运行框架里，结果可能像两套系统。工具怎么暴露，终端输出怎么截断，什么时候检索，失败后如何恢复，完成前是否复核，这些“模型外面”的工程选择，往往决定最终上限。

论文把这一层叫 **harness**。它不是单个 prompt，而是包住模型的可执行程序：提示构建器、检索器、记忆更新器、工具编排器、状态机、评估回调和完成协议。模型像发动机，harness 像传动、导航、仪表和刹车。发动机很强，车也可能开得不稳。

## 1. 为什么普通文本优化不够？

传统文本优化器通常把反馈压缩成标量分数、短模板、当前候选的反思，或者一段由模型生成的摘要。这对短链路 prompt 优化可能够用，但 harness 工程有一个麻烦：因果链很长。

例如，一个“记忆里应该存什么”的决定，可能在几十步后才影响答案；一个“完成前多确认一次”的状态机，可能让 Agent 在已经做完任务后陷入重复验证；一个“让模型清理临时文件”的提示，可能误删评分需要的产物。只看总分，几乎无法知道是哪一层设计导致了回归。

Meta-Harness 的基本判断：

要优化 harness，就不能把失败现场压扁。优化器需要能看到原始代码、完整轨迹、历史分数和工具输出，并能自己决定该查哪段历史、比较哪些候选、改哪部分代码。

## 2. 方法：把经验变成可查询的文件系统

Meta-Harness 的外循环并不复杂：提出候选 harness，跑评估，把候选代码、评估分数、推理轨迹、工具调用、模型输出和状态更新写入一个新目录，然后进入下一轮。真正关键的是反馈接口：历史不是一个 prompt 里的摘要，而是一个不断增长、可搜索、可比较的文件系统。

![Meta-Harness 中文搜索循环图](Meta-Harness 中文搜索循环图)

图 1：Meta-Harness 的关键是完整历史文件系统，而不是压缩反馈。

提议者不是一次性读完整个历史，而是像工程师一样用 `grep`、`cat`、diff 和编辑器选择性读取。在 TerminalBench-2 的搜索运行里，提议者每轮读取文件的中位数是 82 个，其中约 41% 是历史 harness 源码，约 40% 是执行轨迹。也就是说，它真的在跨候选、跨失败地复盘。

论文强调，一次候选评估在困难设置下可能产生高达 1000 万 token 量级的诊断信息，比此前很多文本优化设置的反馈预算高几个数量级。Meta-Harness 没有试图把这些信息塞进上下文窗口，而是让提议者把文件系统当成外部记忆，按需访问。

## 3. 这篇论文的核心创新

| 创新点 | 论文里的含义 | 为什么重要 |
| --- | --- | --- |
| 完整历史访问 | 保存所有候选的源码、分数和执行轨迹。 | 保留长距离因果线索，避免摘要丢掉关键失败证据。 |
| 编码智能体作为提议者 | 使用能调用开发者工具、读写文件和修改代码的 Agent。 | 优化对象是可执行程序，不是单段文本；提议者需要像开发者一样工作。 |
| 非马尔可夫搜索 | 不强制只从最近父代或固定 archive 采样。 | 提议者可以回看任何旧候选，做跨运行对比和因果诊断。 |
| 代码空间正则化 | 搜索完整 harness 程序，包括提示、检索、记忆、工具编排。 | 代码天然鼓励结构化策略，比硬编码答案或脆弱规则更容易审计。 |
| 多目标 Pareto 优化 | 同时考虑准确率和上下文成本。 | 不是盲目堆上下文，而是找到性能与成本之间更好的操作点。 |

这里最值得注意的是第三点：Meta-Harness 没有把进化算法写得很复杂。它故意把外循环做得简单，把“看什么、比较什么、改什么”的决策交给编码智能体。随着编码智能体本身变强，这个外循环理论上也会自然变强。

## 4. 实验：它到底证明了什么？

论文选择了三个跨度很大的场景：在线文本分类、检索增强数学推理、TerminalBench-2 自主编码。它们共享一个特点：harness 的行为会跨多个步骤影响结果，很难靠单步反馈定位问题。

| 任务 | 关键结果 | 应该怎么读 |
| --- | --- | --- |
| 在线文本分类 | 48.6% 平均准确率，ACE 为 40.9%；额外上下文约 45.5K 字符，ACE 为 203.0K。 | 提升不是靠塞更多上下文，而是靠更会选、更会组织证据。 |
| 文本优化器对比 | Meta-Harness 前 4 次评估就达到 OpenEvolve / TTT-Discover 的最终准确率，最终高出 10 点以上。 | 在 harness 优化里，完整历史访问比固定变异策略更关键。 |
| OOD 文本分类 | 9 个未见数据集平均 73.1%，ACE 为 70.2%。 | 发现的策略不是只记住搜索集，而有一定迁移性。 |
| IMO 级数学推理 | 200 道题、5 个保留模型上，平均从 34.1% 到 38.8%，比无检索高 4.7 点。 | 检索是否有用，关键在路由、过滤、重排和插入格式。 |
| TerminalBench-2 | Opus 4.6 上 76.4%，超过 Terminus-KIRA 的 74.7%；Haiku 4.5 上 37.6%。 | 自动搜出的 harness 能在竞争性 Agentic Coding 基准上超过强人工基线。 |

消融实验尤其说明问题。在线文本分类里，如果提议者只能看到分数，中位准确率是 34.6，最佳 41.3；如果看到分数加模型摘要，中位 34.9，最佳 38.7；完整 Meta-Harness 接口的中位数是 50.0，最佳 56.7。也就是说，摘要不仅不能恢复原始轨迹里的信号，有时还会把诊断有用的细节压掉。

文本分类里，强策略叫 **Label-Primed Query**：先显式列出有效标签空间，再为每个标签挑与当前查询相关的覆盖示例，最后加入局部对比对，让模型看到“相似输入为什么属于不同标签”。它更像一个自动发现的分类边界构造器，而不是普通最近邻少样本。

数学推理里，Meta-Harness 没有发明一个新的神经检索模型，而是搜出一个四路 BM25 程序：组合数学、几何、数论、默认路径分别有不同候选数、去重策略、难度重排和返回示例数量。这说明很多实际收益来自“怎么用检索”，而不只是“有没有检索”。

## 5. Artifact：TerminalBench-2 里真正落地了什么？

GitHub artifact 发布的是 TerminalBench-2 上的 Agent scaffold，构建在 KRAFTON AI 的 Terminus-KIRA 和 Harbor 的 Terminus-2 框架之上。README 给出的结果是：Claude Opus 4.6 在 89 个任务、每个任务 5 次试验下，总通过率 76.4%；Easy 100.0、Medium 81.1、Hard 64.7。

![TerminalBench-2 中文 Harness 图](TerminalBench-2 中文 Harness 图)

图 2：裁剪后的论文原图。红色环境快照是 Meta-Harness 发现的新增组件。

最终落地改动很克制：在 Agent 循环开始前，运行一个带超时保护的环境快照命令，把工作目录、`/app` 文件列表、可用语言和版本、包管理器、内存信息注入初始提示。这样，Agent 不必把前 2 到 5 轮花在 `ls`、`which python3`、检查包管理器这类探索上。

这个改动为什么不像“多写一点提示词”？

它是一段可执行运行时逻辑：先观察真实沙盒，再改写初始上下文。对依赖复杂、工具可用性不确定的任务来说，早期错误假设会一路传导；提前把环境事实交给模型，等于减少整条轨迹的偏航概率。

## 6. 最精彩的部分：它会从失败里做因果推理

TerminalBench-2 的搜索轨迹很能说明“完整历史访问”的价值。早期候选尝试修复终端 marker 泄漏、完成确认状态机等问题，但这些结构修复和提示模板改写混在一起，结果反而回退。

到第 3 轮，提议者开始识别混杂因素：退化可能不是结构修复本身造成的，而是清理导向的提示重写破坏了任务完成。随后它隔离变量，只测试较小结构改动；再经历几轮回退后，它总结出“完成流程和提示模板很脆弱”，于是转向纯加法的环境引导。

这不是随机变异。它在做的是工程式归因：哪些候选共享同一个失败因素？哪些修复被另一个改动混淆了？哪些组件最好先别碰？这一类判断，靠单个分数或摘要很难完成。

## 7. 对 Agent 工程的启发

第一，日志不是副产品，而是下一轮优化的训练材料。执行轨迹、工具调用、评分、状态变化和代码版本，最好天然就以机器可查的目录结构保存下来。

第二，skill 文本是搜索质量的杠杆。论文建议 skill 应该规定角色、目录结构、命令、输出格式、禁止行为和目标，但不要把诊断过程写死。好 skill 像边界和仪表，不像遥控器。

第三，搜索集要难，评估要快。若基线已经饱和，外循环没有信号；若评估太慢，搜索成本会吞掉收益。论文建议先构建能暴露失败模式的小搜索集，再自动化完整评估。

第四，轻量验证要放在昂贵评估之前。候选 harness 先通过导入、实例化、小样本调用这类秒级测试，再进入完整基准，能显著降低无效候选的成本。

## 8. 局限：别把它神化

Meta-Harness 依赖强编码智能体作为提议者。论文实验里使用 Claude Code + Opus 4.6；换成更弱或不同风格的提议者，效果如何还需要更多验证。

TerminalBench-2 实验也有特殊性：由于任务少且评估成本高，搜索和最终评估使用同一组 89 个任务。作者做了人工检查和正则审计来降低过拟合风险，但这仍更接近公开竞争基准上的发现问题，而不是严格隐藏测试集泛化证明。

另外，artifact 仓库发布的是被发现的 TerminalBench-2 scaffold，不是完整 Meta-Harness 外循环平台。如果要把方法迁移到自己的领域，还需要自己搭搜索集、评估器、候选验证、日志目录和经验查询接口。

真正值得带走的观点

Agent 的上限不只来自模型，也来自可执行的外部工程层。下一阶段的“提示工程”会更像“运行时工程”：把经验记录好，把评估做扎实，把 harness 当成可以被搜索、审计、回滚和迁移的程序。
