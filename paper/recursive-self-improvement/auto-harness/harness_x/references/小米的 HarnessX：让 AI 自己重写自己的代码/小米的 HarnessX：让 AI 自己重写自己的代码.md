小米最近搞了个挺有意思的东西——**HarnessX**，一个能让 AI Agent 自己重写自己脚手架代码的框架。

这事儿在技术圈子里挺重磅的，主要因为它解决了一个很多团队都在头疼的问题：AI Agent 的 harness（连接大模型和环境的软件脚手架）怎么优化。

## 啥是 Harness？

先搞清楚概念。在 AI 应用里，harness 就是把大模型变成能干活的 Agent 的那层代码。

包括：

* 提示词模板
* 工具集成（API 调用、数据库查询等）
* 记忆管理
* 控制流程
* 错误处理

打个比方：大模型是引擎，harness 就是传动系统、控制系统、转向系统的总和。没有好的 harness，再强的引擎也跑不起来。

## 传统 Harness 有啥问题？

现在企业里的 harness 基本都是手工写的，存在三个核心问题：

**静态且手工制作**

换个模型、加个新工具、换个业务场景，都得手动改代码。没有自动学习和改进机制。

**架构纠缠严重**

提示词、工具封装、重试策略、记忆管理都混在一起。改一处可能坏另一处。复用基本靠复制粘贴。

**Harness 和模型分开优化**

测试时产生的执行轨迹通常被丢弃，不会用来改进模型。模型升级也不自然带来 harness 改进。

这导致一个很尴尬的情况：团队花大钱搞来的执行数据，大部分价值都没捕获到。

## HarnessX 的解决方案

小米的 HarnessX 用了个很巧妙的设计思路：**把 harness 当作"一等对象"**。

### 模块化架构

HarnessX 把 Agent 行为拆解成独立组件：

* 上下文组装
* 记忆管理
* 工具生态系统
* 控制流程
* 可观测性

每个具体行为都是插到 harness 生命周期钩子上的"处理器"。这种模块化结构让系统可以随时替换、添加、删除组件，不会破坏整个流程。

### AEGIS 自动优化引擎

HarnessX 引入了 AEGIS，一个基于执行轨迹的进化引擎。核心思路是把 harness 优化框定为一个强化学习问题。

为了防止强化学习常见的问题，AEGIS 设计了四阶段管道：

1. **Digester**：压缩执行轨迹，找出 Agent 在哪挂了
2. **Planner**：分析这些摘要，探索结构性变化而不只是局部提示词调整
3. **Evolver**：生成代码级别的 harness 编辑，部署前测试确保正确运行
4. **Critic 和 Gate**：Critic 评估编辑是否是奖励黑客攻击，确定性门拒绝任何倒退的更新

## Harness-Model 协同进化

这是 HarnessX 最厉害的地方。

传统的做法是 harness 和模型分开优化：要么只改 harness，要么只训练模型。但这样都会遇到天花板——只改 harness，底层模型推理能力不够，新工具也用不好；只训练模型，harness 永远不会提示模型用高级能力。

HarnessX 让 harness 进化和模型训练交织进行。执行轨迹会被转换成强化学习信号训练基础模型。每次 harness 改进策略，模型就同步学习更好利用这个新策略。

技术上，这通过 **跨 harness GRPO**（Group Relative Policy Optimization）实现。这是用来训练 DeepSeek-R1 这类推理模型的流行 RL 算法。

## 实际效果咋样？

研究人员在五个基准测试上验证了 HarnessX：

* 软件工程
* 多轮客服对话
* 网页导航
* 开放多步推理
* 具身规划

他们设计了两个角色：

* **Meta-Agent**：由 Claude Opus 4.6 驱动，分析日志并编写 harness 代码
* **Task Agents**：运行实际工作流，测试了 Claude Sonnet 4.6、GPT-5.4、Qwen3.5-9B

结果挺亮眼。

### 总体表现

在 15 个模型-基准组合中，HarnessX 在 14 个上提升了性能。平均性能提升 **+14.5%**。

### 弱模型收益最大

开源的 Qwen3.5-9B 在 ALFWorld 具身规划基准上性能跳涨 **+44.0%**，在 SWE-bench Verified 软件工程上提升 **+18.2%**。

### 协同进化的威力

用 harness 进化产生的数据训练基础模型，还能再获得 **+4.7%** 的平均性能提升。

## 实际案例：看 HarnessX 怎么解决问题

### 案例 1：浏览器超时问题

在 GAIA 多步推理基准中，Agent 一直失败，因为用来抓取 Wikipedia 的无头浏览器工具在 JavaScript 密集的前端超时了。

HarnessX 分析执行轨迹，诊断出错误，写了个新工具，直接绕过浏览器查询 MediaWiki API 获取纯文本。替换工具后，失败的任务立马解锁。

### 案例 2：分页循环

在 WebShop 电商测试中，AI Agent 经常陷入分页循环，不断点"下一页"和重构搜索，从不决定购买产品。

HarnessX 没有只调整提示词，而是构建了一个顾问处理器，检测 Agent 何时在重复导航动作。它在上下文中注入警告强制决策，治好了循环行为。

## 我的看法

HarnessX 这个框架挺有启发性的：

**小模型也有春天**

现在行业都在卷模型规模，但 HarnessX 的数据表明，对于很多场景，优化 harness 可能比升级模型更有效。特别是预算有限的团队，先试试 harness 进化，再考虑换更大的模型。

**自动化是趋势**

手工编写和调试 harness 是个慢活儿。让 AI 自己优化自己，这方向肯定是未来的主流。HarnessX 只是开了个头。

**协同进化的威力**

harness 和模型协同进化的思路很有意思。就像软件工程里的持续集成和持续部署，AI 系统也需要持续进化的机制。

**实用价值高**

对于企业来说，这意味着更低的成本、更好的性能、更快的迭代。特别是那些用开源模型的团队，收益可能会很可观。

## 局限性也要看清楚

框架目前依赖强大的模型作为 meta-agent（重写 harness 代码）。在实验中用的是 Claude Opus 这种闭源前沿模型。开源模型进步很快，但能否胜任 meta-agent 角色还有待验证。

另外，如果底层任务模型太弱，无法执行新 harness 提出的复杂工作流，HarnessX 也无法提升 Agent 的整体能力。

## 下一步怎么办？

小米的研究团队计划在未来更新中发布代码。对于开发者来说，这是个值得关注的方向：

1. **短期**：可以借鉴模块化 harness 的设计思路，重构现有系统
2. **中期**：关注开源实现，测试在自家场景的效果
3. **长期**：构建自己的 harness 持续进化能力
