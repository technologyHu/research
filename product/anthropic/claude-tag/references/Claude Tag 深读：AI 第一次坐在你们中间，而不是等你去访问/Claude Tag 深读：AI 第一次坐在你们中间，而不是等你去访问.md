Claude Tag 深读：AI 第一次不是"你去访问的工具"，而是"坐在你们中间的同事"2026 年 6 月 23 日，Anthropic 发布了 Claude Tag——一款把 Claude 直接嵌入 Slack 群聊的 AI 队友产品。本文按"为什么重要 → 怎么运行 → 对国内团队意味着什么"三层展开。一、为什么它值得一个"重大重新设计"Karpathy 给 Claude Tag 定的调子是：这是 LLM UI/UX 的第三次重大重新设计。第一次：LLM 作为网站——你打开 chat.openai.com，去访问它；第二次：LLM 作为桌面/命令行 Agent——Claude Code、Cursor 跟着你的编辑器和终端跑；第三次：LLM 作为持久异步实体——常驻团队协作空间，共享上下文，自主行动。关键词不在"AI"，而在"频道里"。过去的 AI 都是"你去拜访它"，Claude Tag 反转了这个关系：AI 来到你的工作现场，坐在你和同事中间，长期待着，记住一切，自己干活。Anthropic 开发者关系负责人 Alex Albert 说得直白——用 Claude Tag 不像使用工具，更像管理团队：你委派、你监督、你审阅，而不是一个提示一个回答。工作方式从"提示—应答"变成"委派—监督—审阅"。二、一段真实场景：Slack 里的调查现场先不解释概念，看官方文档的一个演示：#platform-eng   38 人  小丹  14:14  结账流程今天上午一直很慢，你们有同感吗？ Leo   14:15  一样。@Claude 帮忙查一下？把延迟和今早的部署做个对比。 Claude 14:15  收到，我来对比部署前后的延迟数据、定位原因。    ✅ 从 Datadog 拉取了 p99 延迟数据   ✅ 将部署 4f2c1 与 main 做了 diff   ✅ 在本地复现了慢查询   ⏳ 正在提交修复的 Pull Request… 几个细节值得咀嚼：Leo 一句话交代下去，Claude 自己决定执行顺序（拉指标 → 看代码 → 复现 → 提 PR）；38 个人都能看到调查过程；Leo 去开会了，小丹发现方向不对可以直接在同一话题里回复指正，session 会把她的指令融合进去，无需重开一轮。这就是 Claude Tag 想干的事：Session 不再属于用户，而是属于"话题"。三、五层架构，一个都不能少Claude Tag 的运行机制可以拆成五层，每一层都对应一个"AI 队友"必须回答的问题：层能力回答的问题Multiplayer一频道一个共享 Claude谁在看？——群聊内所有人共享同一段对话、同一组行动Persistent Memory频道级持久记忆它知道什么？——按频道积累团队的隐性知识（代码约定、架构决策、模块负责人）Async Delegation跨小时/跨天异步执行它能干什么？——委派后自行推进，跟进未解决的线程Ambient Mode环境感知主动行为它主动不主动？——监控频道活动、跨频道关联、主动跟进被搁置的任务Scoped Governance按频道隔离的治理层谁管得了它？——每个频道独立身份、独立工具授权、独立 Token 预算五层缺一，"队友感"就会塌方——共享身份没有持久记忆，就还是一次性问答；有记忆没有异步执行，就还是被动响应；能异步不能主动感知，就还是"你想起来才用它"；主动感知没有治理，就变成"办公室里那个总在你身后看屏幕的同事"。四、Session 属于"话题"，权限跟着"频道"走Session 模型是 Claude Tag 最关键的架构选择。传统 AI 聊天：一个用户 → 一次对话 → 一个 Session → 结束就没了。Claude Tag：一个话题 → 一个持久 Session → 群聊内所有人共享 → 回复即继续 → 空闲后仍可激活。这一个决定，cascade 出整个产品的其余部分：权限跟频道走，不跟人走——Claude 能访问什么，取决于你在哪个群聊，而非你是谁；记忆按频道组织——公开频道记忆共享，私密频道记忆隔离，私密转公开时旧记忆不跟着迁移（防止意外泄露）；费用记在组织账上——群聊里的操作走组织账单，私信里的操作走个人 claude.ai。围绕这个 Session 模型，Anthropic 定义了一整套术语（不完整但够用）：Access Bundle：管理员配置的一组连接、仓库权限和规则；Scope：Bundle 生效的三个层级——组织默认 → 工作区 → 单个群聊；Connection（组织级）vs Connector（个人级）：前者是 Agent Identity 名下的组织凭证，只在群聊里用；后者是你个人 claude.ai 的连接器，只在私信里用；Agent Proxy：网络中间层，在 Claude 的出站请求里注入凭证——模型和沙箱永远拿不到 Key；Routine：Claude 自动运行的定时或一次性任务，用群聊的 Connection，不用创建者个人的凭证；Channel Memory：Claude 在群聊中积累的记忆。五、三层安全架构：架构级硬隔离，不是软约束这是 Claude Tag 最值得国内团队学习的一段。安全不是"在 prompt 里告诉模型不要泄露密钥"，而是架构级的硬隔离。┌──────────────────┐   ┌────────────────────┐   ┌────────────┐ │    沙箱 Session    │──▶│    Agent Proxy      │──▶│  外部系统   │ │                   │   │                     │   │ └────────────────────┘   └────────────┘ 第一层 沙箱（零信任）：模型不直接持有任何 API Key、Token、密码；第二层 Agent Proxy：出站请求先到 Proxy，Proxy 做两件事——Rule 匹配（目标主机/路径/方法是否在规则内）+ 凭证注入（从与 Proxy 分离的独立凭证存储中取出并注入）。保存后的凭证不再展示给任何人，包括管理员。第三层 外部系统：使用 Agent 自己的服务账号访问，走的是 Agent Identity，不是任何一个人的账号。出站流量默认 deny-all。不在白名单里的主机，Claude 连 DNS 解析都做不到——不是"你不应该访问"的软指令，而是网络层面的硬隔离。对比很多团队沙箱目前把 IAM token 存在 Mongo 里、沙箱进程直接持有凭证的做法，差距不小。Agent Proxy 模式值得抄。六、Ambient Mode：能力最抢眼，也最有争议Ambient Mode 让 Claude 从"被动响应"转向"主动发起"——监控频道活动、跨频道关联洞察、跟进搁置的任务。这是竞争力来源，也是治理风险来源。社区里有一句评论很到位："Ambient 这个词做了很多工作——它听起来很轻柔、很环境化，但背后是持续监控、持续推断、有时还有未经请求的干预。"它同时是"生产力工具"和"监控系统的温和面孔"，两种可能性住在同一个 feature 里。学术上还有一层张力：Weiser 1991 年定义的 ambient 强调技术消失于环境，用户不再感知；Claude Tag 的 Ambient Mode 是显式在场的主动干预——两者概念上并不重合。这条路怎么走，目前还没有成熟答案。七、和竞品比：三位一体的护城河竞品对比：维度Claude TagCopilot / Work IQSalesforce SlackbotGlean飞书智能伙伴存在方式常驻频道，共享身份，ambient Copilot一人一 Slackbot一人一搜索会话一群一伙伴，上下文不持久四组核心差异可以归纳成一句话：持久记忆 + 多人共享 + 主动介入 = 三位一体。缺一，"队友感"就塌方。持久记忆让 AI 越用越懂你，迁移成本随时间指数增长；多人共享让 AI 从个人助手变成组织资产，PM 和工程师看到同一个 Claude 的回复，上下文对齐替代重复沟通；主动介入让 AI 从"你想起来才用它"变成"它想起来就提醒你"，填补人类注意力的缝隙。八、Anthropic 自己的用法：65% 的代码Anthropic 是 Claude Tag 的首批深度用户。工程频道授权 GitHub+CI+Jira，产品频道授权 Jira+Metabase，记忆和权限严格隔离。工程师在频道里 @Claude 委派编码任务，Claude 拆解成阶段、异步执行、线程更新进展。官方数据：产品团队 65% 的代码由 Claude Tag 的内部版本创作，65% 的 Pull Request 由它开启。这个数字要审慎看——"65% 的代码"没交代计量口径（行数？字符数？函数数？），"产品团队"的组成也没披露，全部是 Anthropic 自报数据。但它至少说明一件事：如果自家团队都不重度用的产品，对外说服力会大打折扣。比数字更值得抄的是他们的工作方式转变——"我们花在并行委派任务给多个 Claude 的时间，远远多于亲自执行这些任务。"从"人做事"变成"人分配事、Claude 做事、人审核结果"。九、对国内团队意味着什么Claude Tag 只在 Slack 上跑，Slack 在中国企业几乎不存在。所以对我们真正的问题是：这套形态搬到飞书/钉钉/企业微信上，缺什么？答案——底座已有骨骼，缺的是三位一体的灵魂：持久记忆 + 共享身份 + Ambient 主动介入。飞书 CLI 已覆盖 18 个业务域、200+ 命令、26 个 AI Agent Skill；钉钉 CLI 在业务域也有广泛覆盖；企业微信具备 AI 集成能力。API 通路都已铺好，差的不是接入，是产品原语：一个频道一个共享的 Agent、按频道积累的持久记忆、跨频道的凭证边界、可审计的 Agent Proxy、可关掉的主动介入。已有社区尝试：开发者 zarazhangrui 开源了 lark-coding-agent-bridge（npm 包 lark-channel-bridge），把本地 Claude Code 桥接到飞书——手机端对话、多 session 管理、飞书卡片交互。但和 Claude Tag 的差距在四个方向都清晰可见：能力Claude Tag (Slack)lark-bridge (飞书)持久上下文记忆频道级长期积累Session 级，断开丢失Ambient 主动监控 + 主动推送仅被动 @bot 响应多人共享上下文一频道一 Claude多 session 隔离异步任务追踪自动跟进搁置任务无谁先在自己的底座上把三位一体做起来，谁就占据下一代工作方式的定义权。十、几个还没被回答的问题持久化的 Prompt Injection：无状态的 Chatbot 被注入了，坏的只是这一次对话；Claude Tag 有跨天记忆和 MCP 工具连接，一次成功注入可能污染整个频道的长期记忆，后续所有交互都被影响。传统安全测试方法论可能不再适用。Ambient 的治理张力：主动介入是能力，也是"温和的监控"。什么时候介入、以什么频率介入、误触发的成本谁承担——目前没有成熟答案。供应商锁定的深度：当团队的组织记忆、代码知识、协作规则都沉淀在一个供应商的系统里，迁移成本已经不是数据搬家，而是认知资本的断裂。组合惩罚（Vaccaro et al., 2024）：Nature Human Behaviour 上的元分析显示，人机组合在多数任务中平均表现不如人或 AI 单独工作。多人共享一个 AI 是否会放大而不是缓解这个问题？现在没人有确定答案。三条外带如果只带三条走：一，Session 模型是根——一旦决定 Session 属于"话题"而不是"用户"，权限、记忆、费用、审计的所有下游设计都会跟着重写。这是最值得先想清楚的一步。二，Agent Proxy 是标杆——凭证不进模型、不进沙箱，只在网络边界注入；出站默认 deny-all；凭证存储与 Proxy 分离；保存后的凭证不再展示给任何人。这套架构可以直接抄。三，三位一体是护城河——持久记忆、共享身份、主动介入。国内底座缺的不是入口，缺的是这三样一起做起来的产品原语。AI 从"工具"变成"同事"这件事，已经不是要不要发生，而是在哪个平台上先发生。
